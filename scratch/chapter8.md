# Chapter 8: Appendix & Expected Graduation Committee Questions

This chapter serves as a comprehensive technical appendix for the **SOLIX Platform (SOL Data Agent)**. It includes a glossary of technical terms and detailed answers to strategic questions commonly asked by academic graduation review committees and technical evaluators.

## 8.1 Technical Glossary

*   **FastAPI**: A high-performance, asynchronous web framework for building APIs with Python 3.8+, based on standard Python type hints.
*   **Polars**: A lightning-fast DataFrame library written in Rust, designed for parallel, memory-efficient data processing.
*   **Pandas**: A widely used Python library for data manipulation and analysis, used by the platform to apply in-memory cell mutations.
*   **Lazy Evaluation**: An execution strategy where data operations are not computed immediately, but are instead logged as a query plan to be optimized and executed later. Used by Polars to profile datasets efficiently.
*   **JSON Web Token (JWT)**: An open standard (RFC 7519) that defines a compact, self-contained way to securely transmit information between parties as a JSON object.
*   **Secure One-Time Password (OTP)**: A temporary passcode sent to verify an identity. Hash values of these codes are stored using SHA-256 to prevent unauthorized use.
*   **ReAct Agent (Reasoning and Action)**: An AI agent architecture that combines reasoning (thinking steps) with action execution (running tools) to solve complex user tasks.
*   **Abstract Syntax Tree (AST)**: A tree representation of the abstract syntactic structure of source code. Used to analyze Python scripts for safety before they are executed.
*   **Vector Database (Qdrant)**: A database designed to store, index, and search high-dimensional vector embeddings, enabling fast semantic similarity queries.
*   **Dense Embeddings**: Numerical vector representations of text where semantically similar phrases are located close to each other in vector space.
*   **Generative Adversarial Network (GAN)**: A class of machine learning frameworks where two neural networks (a Generator and a Discriminator) contest with each other, used to generate synthetic tabular data (e.g. CTGAN).
*   **Server-Sent Events (SSE)**: A server push technology enabling a client to receive automatic, real-time updates from a server over a HTTP connection, used to stream AutoML progress.

## 8.2 Expected Oral Defense Questions and Model Answers

The following questions cover the design decisions, trade-offs, and security features of the SOLIX platform.

### Question 1: Why did you choose a hybrid Pandas/Polars execution model? Why not use Pandas for everything?
**Answer:** "Pandas is excellent for in-memory mutations, spelling updates, and cell editing, but it loads the entire dataset into memory (RAM). This can lead to Out of Memory (OOM) errors or slow performance when handling large files. Polars, written in Rust, supports lazy execution and parallel processing. We use Polars to scan, profile, and evaluate incoming datasets without loading them entirely into memory. This hybrid approach uses Polars for fast, low-memory file loading and analysis, and Pandas for targeted data updates."

### Question 2: How does the AST parser safeguard against malicious code execution in the voice sandbox?
**Answer:** "The ReAct agent generates Python code to edit datasets. To prevent the execution of malicious scripts (e.g., deleting system files or accessing env credentials), we parse the code into an Abstract Syntax Tree (AST) before running it. A custom `_SecurityVisitor` class traverses the tree nodes. If it finds forbidden imports (like `os`, `sys`, `subprocess`, or `shutil`) or restricted built-in functions (like `eval`, `exec`, `open`, `globals`, or `locals`), it blocks execution and returns a safety error. This ensures code execution remains safe."

### Question 3: How does the system handle uploading large files (e.g., 2GB CSV) without crashing?
**Answer:** "The platform uses three main strategies:
1.  **Chunked Uploads**: The file upload endpoint (`/api/upload`) reads incoming files in 1MB chunks and writes them directly to disk, avoiding RAM congestion.
2.  **Parquet Conversion**: Raw uploaded files are converted into Apache Parquet format on disk. Parquet uses columnar storage, which speeds up queries and reduces file sizes.
3.  **Lazy Scanning**: We use Polars lazy frames to read files from disk only when needed, avoiding loading the entire dataset into memory."

### Question 4: Why do you need Qdrant Vector DB for the Dataset Advisor instead of SQL `LIKE` queries?
**Answer:** "SQL `LIKE` queries only match exact substrings. If a user searches for 'automobile logs', a SQL query will miss datasets tagged with 'car history' or 'vehicle records'. Qdrant stores high-dimensional dense vector embeddings generated by a multilingual sentence-transformer model. This allows the Dataset Advisor to understand the meaning behind queries (e.g., in English or Arabic) and return relevant datasets based on semantic similarity rather than exact spelling matches."

### Question 5: How does the Undo/Redo state stack prevent RAM bloat on large datasets?
**Answer:** "If we kept multiple versions of a large dataset in memory to support undo and redo operations, the application would quickly run out of memory. To avoid this, our state manager (`_store`) saves a versioned Parquet snapshot to disk (e.g. `temp_snapshots/{dataset_id}_v{version}.parquet`) every time a dataset is cleaned or modified. The session only keeps track of the active version index and file paths. When a user calls undo or redo, the application loads the corresponding file from disk, which keeps memory usage low."

### Question 6: What is the difference between Gaussian Copula, TVAE, and CTGAN models in the Synthetic Data Studio?
**Answer:** "These models are used to generate synthetic tabular data:
*   **Gaussian Copula**: A statistical model that matches the distribution and correlation between columns. It is fast and works well for simple datasets, but struggles with complex, non-linear relationships.
*   **TVAE (Tabular Variational Autoencoder)**: A neural network that compresses data into a lower-dimensional space and reconstructs it. It trains quickly and handles complex data patterns well.
*   **CTGAN (Conditional Tabular GAN)**: A generative adversarial network designed specifically for tabular data. It is excellent at generating realistic data for highly imbalanced categorical columns, though it requires more time and CPU/GPU resources to train."

### Question 7: How did you solve external API key and system dependency requirements for Windows environments?
**Answer:** "We configured a `.env` template file containing keys for external APIs (like Groq, ElevenLabs, and Kaggle) and local database paths. For system dependencies like Tesseract OCR, the backend checks standard installation directories (e.g., `C:\Program Files\Tesseract-OCR\tesseract.exe`). If the binary is not found, the system disables OCR features and falls back to a PDF text extractor instead of crashing, ensuring the application remains usable."

### Question 8: How do you achieve low-latency voice feedback in the Arabic Voice Copilot?
**Answer:** "We use three main techniques:
1.  **Gatekeeper Classification**: The gatekeeper agent immediately filters out non-command queries, bypassing code generation steps.
2.  **Arabic Brevity Law**: The agent is prompted to limit voice responses to a maximum of 25 words.
3.  **Streamed Audio**: The TTS endpoint communicates with ElevenLabs to stream the synthesized audio in chunks as it is generated, rather than waiting for the entire audio file to process."

### Question 9: Why did you use two cookies for JWT instead of standard HTML localStorage?
**Answer:** "Storing JWTs in `localStorage` makes them vulnerable to Cross-Site Scripting (XSS) attacks; any malicious JavaScript running in the browser can access the storage. We store the authentication token in a secure, `HttpOnly` cookie named `access_token`, which cannot be accessed by client-side scripts. We also set a second, standard cookie named `sol_auth_token` that contains basic user details. This allows client-side scripts to verify the login state and update the UI without risking the security of the main authentication token."

### Question 10: How does the AutoML Studio stream real-time training progress to the browser?
**Answer:** "Instead of making the browser poll the server repeatedly, which causes network overhead, we use Server-Sent Events (SSE) via the `/api/automl/train` endpoint. This establishes a persistent, one-way HTTP connection. As the backend pre-processes data, trains models, and runs evaluations, it pushes structured JSON updates (e.g., `{"percent": 45, "step": "Training LightGBM"}`) to the client, allowing the frontend dashboard to update its progress bar in real-time."
